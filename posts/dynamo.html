01 / Why Dynamo Exists

Serving large language models is constrained by memory bandwidth and scheduling

In autoregressive inference, a significant portion of latency comes from reading the KV cache from GPU memory at every decoding step. As model sizes grow beyond tens of billions of parameters, the KV cache can exceed the capacity of a single GPU, making interconnect bandwidth and placement decisions critical.

At typical serving batch sizes, inference is often limited more by memory bandwidth than raw compute. The workload also has two distinct phases: prefill, which processes the input prompt and is compute-heavy, and decode, which generates tokens sequentially and is more memory-bound. Most existing serving systems prior to 2025 execute both phases on the same hardware, which leads to suboptimal utilization for at least one of them.

Dynamo, introduced at GTC 2025 and reaching 1.0 in March 2026, addresses this by separating prefill and decode into independently scheduled workloads. This allows each phase to be mapped differently across GPUs, taking into account parallelism strategy, memory behavior, and hardware topology.

Key idea: Disaggregated serving separates prefill and decode into different compute pools, allowing each stage to be scheduled and scaled independently.

02 / Architecture Internals

Four subsystems that differentiate Dynamo from a traditional serving stack

dynamo 1.0 — component topology

a. Disaggregated Prefill / Decode

The key architectural change is the separation of prefill and decode into distinct GPU pools. Each pool is configured with different parallelism strategies: prefill is compute-bound and typically uses higher tensor parallelism, while decode is memory-bound and benefits from configurations optimized for KV access and MoE routing.

KV cache state is transferred between pools using NIXL, an asynchronous RDMA-based transport layer for KV movement. This avoids synchronous blocking during handoff between prefill and decode, which would otherwise reduce throughput in a disaggregated setup.

b. Smart Router with KV-Aware Routing

The router prioritizes KV cache reuse rather than acting purely as a load balancer. Requests are fingerprinted by prefix, and when a matching cache entry exists, they are directed to a worker with warm KV state. This reduces recomputation of shared prefixes and improves time-to-first-token (TTFT), especially in workloads with repeated system prompts or similar contexts.

Integration with higher-level orchestration layers allows routing decisions to incorporate cache locality and request-level hints.

c. AIConfigurator — Deployment Automation Layer

AIConfigurator takes model specifications, hardware constraints, and SLA targets as input and produces a deployment configuration. It determines GPU partitioning, parallelism degrees, and batch sizing through profiling and heuristic-based optimization.

The output is a deployable configuration that encodes prefill/decode allocation, tensor and expert parallelism settings, and KV cache capacity planning.

d. KV Block Manager with Multi-Tier Offload

KV cache is managed as fixed-size blocks with prefix sharing where possible. Under memory pressure, blocks can be evicted to external storage systems such as object stores and later restored for session continuation.

This design trades increased reactivation latency for higher effective GPU memory capacity, enabling longer context retention and higher concurrency under constrained hardware budgets.

04 / Hardware Topology

Why NVL72 is a strong fit for disaggregated LLM inference

Platform
NVL72

NVL72 systems consist of 72 GPUs (GB200/GB300-class) connected via a high-bandwidth NVLink fabric. This topology provides uniform intra-rack communication bandwidth, which is important for disaggregated inference where KV state may be transferred between prefill and decode pools.

In this setup, scheduling decisions can exploit the NVLink domain to reduce cross-node traffic. KV movement between partitions is handled through a high-throughput transport layer such as NIXL.

72 GPUs · NVLink fabric · high-bandwidth intra-rack communication
Next Generation
Rubin GPUs

Rubin-class GPUs are expected to continue increasing HBM bandwidth and compute throughput relative to prior generations. In disaggregated inference systems, improvements in memory bandwidth typically translate into larger decode batch sizes and improved throughput under memory-bound workloads.

Higher HBM bandwidth · improved decode scalability
CPU Backend
Vera CPUs

In NVL-class systems, CPU-side components handle orchestration tasks such as request routing, tokenization, and KV metadata management. This allows GPUs to remain focused on tensor execution while control-plane work is offloaded.

Some deployments also explore CPU-side tokenization and pipeline overlap, where CPU preparation of the next batch can run concurrently with GPU execution of the current step, reducing idle time in the inference loop.

Orchestration · control-plane offload · pipeline overlap

05 / Request Lifecycle

How a single request moves through a disaggregated inference system

01
Admission & Prefix Lookup router
The request enters the router and is indexed by a prefix hash. If a worker has a matching KV cache entry, the request is routed to reuse that state. On a miss, it is assigned to a prefill worker based on load and capacity.
02
Prefill Execution prefill pool
The full input sequence is processed in a compute-heavy forward pass. Prefill workers typically run with higher tensor parallelism to maximize throughput. CPU-side components may prepare subsequent requests in parallel with GPU execution.
03
KV State Transfer NIXL
After the first token is produced, the KV cache state is transferred asynchronously to a decode worker using an RDMA-capable transport layer. This allows the prefill worker to release resources immediately.
04
Decode Loop decode pool
Autoregressive generation proceeds token by token. Decode workers are optimized for memory-bound execution and may use expert parallelism for MoE models. KV cache is updated incrementally at each step, and tokens are streamed to the client.
05
KV Eviction & Offload KV Block Manager
When GPU memory pressure increases, KV cache blocks may be evicted to external storage tiers such as object storage. On later resumption, blocks can be reloaded, trading reactivation latency for higher effective memory capacity.

06 / Positioning vs Existing Systems

How Dynamo fits into the existing LLM serving stack

vs. vLLM
vLLM introduced paged attention and continuous batching for single-node serving efficiency. Dynamo operates at a higher orchestration layer, coordinating multiple engines and nodes, and can use vLLM as one of its execution backends while adding disaggregated serving and KV-aware routing.
vs. TensorRT-LLM
TensorRT-LLM focuses on optimizing individual model execution through graph-level and kernel-level compilation. Dynamo operates above this layer, handling scheduling, routing, and multi-node execution, and can integrate TensorRT-LLM as a backend engine.
vs. SGLang
SGLang provides structured generation and KV cache optimization features such as HiCache. Dynamo overlaps in some of these areas but focuses on system-wide orchestration across multiple nodes and workloads, and can interoperate with similar KV transfer mechanisms in distributed setups.

Rather than replacing systems such as vLLM, TensorRT-LLM, or SGLang, it coordinates their execution across nodes with explicit attention to routing, scheduling, and KV cache movement.

07 / When Dynamo Underperforms

Practical limits of disaggregated inference systems

Disaggregated serving introduces additional overhead from KV transfer between prefill and decode stages. At low request rates or short sequence lengths, a co-located execution model can be more efficient due to reduced communication cost.

Deployment configuration and profiling can introduce startup latency. This is more suitable for steady-state production workloads than for rapidly changing or cold-start-heavy environments.

The benefit of KV-aware routing depends on prefix reuse in the workload. If requests are diverse with low prefix overlap, cache hit rates drop and routing behaves closer to standard load balancing without significant acceleration.

Fault tolerance mechanisms in early versions of distributed inference systems are typically less mature than single-node serving stacks. While production deployments exist, failure recovery paths may still be evolving.
