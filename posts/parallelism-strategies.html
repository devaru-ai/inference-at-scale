Parallelism Strategies and Mixture-of-Experts in LLM Scaling

Tradeoffs in data, tensor, and pipeline parallelism, plus the compute and communication costs of Mixture-of-Experts.

Apr 2026 18 min read

01. Why parallelism exists

The idea that "more GPUs = faster training" stops being true at large scale. The first constraint is not compute — it is memory.

For a 70B parameter model in BF16, weights alone require ~140 GB. During training, optimizer states (e.g., Adam’s first and second moments in FP32) add roughly 2× parameter size, pushing total memory requirements well beyond 700 GB. A single A100 provides 80 GB of HBM, so the model cannot reside on one device.

This creates two operating regimes:

  • Memory-bound: performance is limited by HBM capacity and bandwidth. Compute units stall waiting for data movement.
  • Compute-bound: arithmetic throughput is the bottleneck. Memory system is saturated but sufficient.

Large batch training trends toward compute-bound execution. Small batches with large models trend toward memory-bound execution. Most real training workloads oscillate between both regimes, which is why parallelism strategy is critical.

Core objective of parallelism: partition model state and computation across devices such that memory fits, compute stays utilized, and communication overhead does not dominate step time.

02. Data parallelism, tensor parallelism, and pipeline parallelism

Modern distributed training relies on three primary parallelism axes, each addressing a different bottleneck.

Data parallelism (DP)

Data parallelism replicates the full model on each GPU and partitions input batches across replicas. After the backward pass, gradients are synchronized using an all-reduce operation. Ring all-reduce is commonly used, where devices exchange partial gradients in a pipelined ring topology.

Communication cost scales as 2(N-1)/N × model_size per step. While bandwidth-efficient, gradient synchronization becomes expensive at scale, especially for large parameter counts.

DP limitation: it assumes the model fits on a single device. If it does not, DP alone is insufficient regardless of batch scaling.

Tensor parallelism (TP)

Tensor parallelism shards individual weight matrices across devices. In transformer feed-forward and attention layers, matrix multiplications are partitioned column-wise and row-wise across GPUs. Each device computes partial results on a subset of weights.

Because partial activations must be combined, TP introduces frequent all-gather and all-reduce operations at nearly every layer. This creates frequent communication between GPUs.

TP performs well within a single node using NVLink (~600 GB/s), but degrades significantly across nodes due to InfiniBand bandwidth constraints (25–400 Gb/s).

Practical constraint: TP is typically bounded by intra-node topology. 8-way TP within a DGX-class node is standard; cross-node TP is usually inefficient.

Pipeline parallelism (PP)

Pipeline parallelism partitions model depth across devices. Each GPU executes a contiguous block of layers, passing activations forward in sequence.

The key inefficiency is pipeline “bubbles”: devices remain idle while waiting for upstream computation. With k pipeline stages, bubble utilization is (k-1)/k, which becomes severe at shallow microbatching.

Micro-batching mitigates this by splitting input batches into m smaller chunks. This allows overlap of forward and backward passes across stages, reducing bubble overhead to (k-1)/(m+k-1). The tradeoff is increased activation memory and scheduling complexity.

03. Hybrid parallelism: what real systems actually do

No single parallelism strategy scales to 100B+ parameter training. Production systems combine multiple axes, and the setup is usually driven by hardware topology more than algorithmic preference.

  • Tensor parallelism (TP) within a node — NVLink/NVSwitch enables high-bandwidth, low-latency collectives. 8-way TP across 8 GPUs in a DGX-class node is typical.
  • Pipeline parallelism (PP) across nodes — inter-node bandwidth is significantly lower, so models are split by depth. Only activations are communicated at stage boundaries, making PP relatively communication-efficient over InfiniBand.
  • Data parallelism (DP) across replicas — full TP+PP partitions are replicated across node groups. Gradients are synchronized via all-reduce, typically overlapped with backward computation to hide latency.
Frameworks such as Megatron-LM formalize this as 3D parallelism. Each axis (TP, PP, DP) is independently tunable, and the configuration directly determines MFU (model FLOPs utilization), which is the dominant efficiency metric in large-scale training.

04. Why dense models stop scaling — and what MoE changes

In dense transformers, compute grows with parameter count: every token activates the full parameter set. This makes capacity scaling linearly expensive in FLOPs.

The key observation is conditional computation. Different tokens activate different internal substructures — code, natural language, and math exhibit different latent routing behaviors. This motivates sparse activation.

Mixture-of-Experts (MoE) decouples parameter count from compute by activating only a subset of parameters per token.

05. MoE: mechanism and failure modes

Forward computation

MoE replaces the feed-forward network in transformer blocks with a set of N expert MLPs plus a routing function. For each token, the router selects top-k experts (typically k=1 or k=2). Only selected experts execute, and outputs are aggregated via a weighted combination.

This enables large parameter counts with bounded per-token compute. For example, a 47B-parameter MoE model may activate only ~13B parameters per token depending on routing policy.

Key property: capacity scales with number of experts, while FLOPs scale with k, not N.

06. System-level mitigations

MoE training stability depends on explicit correction mechanisms that counteract routing dynamics and system imbalance.

Load-balancing loss introduces an auxiliary regularizer that penalizes skewed expert utilization, enforcing entropy in routing distributions.

Capacity constraints bound per-expert token assignment using a fixed capacity factor. Overflow tokens are dropped or rerouted, trading completeness for bounded compute.

Expert parallelism distributes experts across devices and makes all-to-all communication the primary synchronization primitive, effectively replacing gradient-centric communication with routing-centric communication.

Together, these mechanisms extend the TP/PP/DP stack with a fourth axis: expert parallelism (EP), completing modern large-scale training systems.

When NOT to use MoE: bandwidth-constrained clusters, latency-sensitive serving (routing overhead adds up), small-scale training (complexity not worth it below ~50B equivalent), or when you lack the engineering bandwidth to tune load balancing carefully.

08. Cluster topology and system constraints

Parallelism design is constrained primarily by hardware topology rather than algorithmic preference. Identical GPU counts can yield materially different optimal configurations depending on interconnect bandwidth and latency characteristics.

  • Intra-node interconnect (NVLink / NVSwitch): ~600 GB/s bidirectional bandwidth with low latency. This makes frequent collective operations (TP all-gathers, reductions) viable. It also allows intra-node expert communication in MoE systems without immediate bandwidth saturation.
  • Inter-node interconnect (InfiniBand): ~25–400 Gb/s depending on generation and configuration. This is several orders of magnitude slower in effective collective throughput. Tensor parallelism across nodes is typically avoided. Pipeline parallelism is preferred because it minimizes communication frequency and confines transfers to activation boundaries.

MoE systems are especially sensitive to topology. The routing all-to-all introduces quadratic communication patterns across participating devices. If experts are distributed without regard to network locality, routing traffic saturates inter-node links quickly.

As a result, expert placement is a first-order design variable. Production systems frequently co-locate experts that are statistically co-activated to reduce cross-node routing traffic. This transforms routing from a uniform all-to-all into a partially localized exchange pattern.

Key structural shift: dense training is dominated by gradient synchronization (DP all-reduce), while MoE training is dominated by token routing (EP all-to-all). The network bottleneck depends entirely on which regime the system operates in.

09. System composition for large-scale MoE training

A realistic large-scale configuration (100B+ parameter class MoE) composes all four axes of parallelism simultaneously. Each axis is mapped to a different layer of the hardware hierarchy.

  • Tensor parallelism (TP) within nodes: 8-way sharding across NVLink-connected GPUs. Used for intra-layer matrix multiplication parallelism.
  • Pipeline parallelism (PP) across nodes: model depth partitioned across node groups. Micro-batching is required to amortize pipeline bubbles.
  • Expert parallelism (EP) across node groups: experts are distributed across devices; routing triggers all-to-all communication as the dominant inter-node operation.
  • Data parallelism (DP) across replicas: full TP+PP+EP stacks are replicated, with gradient synchronization via all-reduce overlapping backward computation.

The system combines four different communication patterns: TP is high-frequency intra-node communication, PP is low-frequency sequential activation passing, EP is bursty all-to-all traffic, and DP is periodic global synchronization.

MFU sensitivity: inefficiencies in any single axis reduce overall utilization. A system at 55% MFU versus 40% MFU reflects not just kernel optimization, but global balance across all four communication domains.

At this scale, performance differences are bottlenecked by topology alignment rather than algorithmic choice. A poor match between parallelism strategy and network topology can significantly reduce throughput.

10. Conclusion

Parallelism exists to decompose memory and compute constraints across devices. MoE extends this by decoupling parameter count from per-token compute, but introduces a more complex communication surface dominated by routing dynamics.

In practice, performance is determined by whether the parallelism strategy matches the underlying hardware topology. When it does, utilization is high. When it does not, local optimizations usually cannot recover the lost throughput.
